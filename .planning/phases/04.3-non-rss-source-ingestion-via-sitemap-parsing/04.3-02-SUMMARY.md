## Decisions Made

- Manual migration applied via Neon HTTP (TCP/5432 blocked in local environment). Created migration SQL file, applied ALTER TABLE statements using `neon()` client, and recorded entry in `_prisma_migrations` table to keep Prisma migration state consistent.
- Existing RSS sources need no `sourceType` in seed.ts: the schema default `"rss"` is set on the column, and `upsert({ update: {} })` does not overwrite existing rows.
- Used `findMany({ select: ... })` to pass only the fields required by `FetchSource` type, keeping the type contract clean and avoiding over-fetching.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Applied migration via Neon HTTP instead of prisma migrate dev**
- **Found during:** Task 1
- **Issue:** `prisma migrate dev` requires TCP/5432 connection. Environment blocks TCP/5432; only Neon HTTP API is accessible. `.env.local` also has `DATABASE_URL_UNPOOLED` with `&` characters preventing shell export.
- **Fix:** Created migration SQL file manually, applied via `neon()` CJS inline script, recorded in `_prisma_migrations` table. This matches the approach used in Phase 04.1.
- **Files modified:** prisma/migrations/20260325200000_add_source_type_fields/migration.sql
- **Commit:** 60af1b4

**2. [Rule 3 - Blocking] Used direct neon() SQL to insert seed rows instead of pnpm seed**
- **Found during:** Task 1
- **Issue:** `pnpm seed` runs `tsx prisma/seed.ts` which uses `new PrismaClient()` requiring TCP/5432. Shell `source .env.local` fails because URLs contain `&` breaking argument parsing.
- **Fix:** Inserted the two new sitemap sources directly via Neon HTTP in a CJS inline script with tagged template literals. The 8 existing RSS sources already exist in the DB (no change needed).
- **Verification:** Confirmed both rows present via SELECT query before and after.
