## Summary

This phase is a no-behavior-change cleanup pass across the entire codebase, organized in two plans: lib+API routes first, then React components. The work is fully scoped by the CONTEXT.md decisions -- no new features, no schema changes, no new tests required.

The codebase has matured significantly since the initial 2026-03-19 analysis. The `any`-typed where clause in the articles API has already been fixed (now uses `Prisma.ArticleWhereInput`). Remaining `any` types are confined to `thumbnail.ts`, `rss.ts` line 61 (rss-parser custom field access), and `chat/route.ts` (Zod `z.any()` for AI SDK message parts). The PATCH endpoint for articles lacks body validation via Zod. SourceList delete has no `response.ok` check. FeedTable toggle rollback is already implemented correctly.

The React component audit via the two skills (react-best-practices, composition-patterns) surfaces one HIGH-impact issue: `lucide-react` is imported from the barrel without `optimizePackageImports` in `next.config.ts`. Boolean prop usage across chat and briefing components is modest and mostly appropriate; no compound-component refactors are required. The `z.any()` in the chat schema for AI SDK message parts is the pragmatic choice given the SDK's polymorphic part types -- narrowing it would require importing internal SDK types.

**Primary recommendation:** Execute Plan 1 (lib + API security/type fixes) then Plan 2 (component audit + `optimizePackageImports`). Both plans are low-risk with no behavior changes.

## Architecture Patterns

### Pattern 1: Zod body validation in Next.js route handlers
**What:** Parse and validate request body with a Zod schema before using any field.
**When to use:** Any PATCH/POST endpoint that reads `request.json()`.
**Example:**
```typescript
// src/app/api/articles/[id]/route.ts
import { z } from 'zod';

const PatchBodySchema = z.object({
  isRead: z.boolean(),
});

export async function PATCH(request: NextRequest, ...) {
  const body = await request.json();
  const parsed = PatchBodySchema.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ error: 'Invalid body' }, { status: 400 });
  }
  const article = await prisma.article.update({
    where: { id: articleId },
    data: { isRead: parsed.data.isRead },
  });
  return NextResponse.json(article);
}
```

### Pattern 2: Response.ok check with rollback (already correct in FeedTable)
**What:** Optimistic update applied immediately; rolled back in catch block if server rejects.
**Status:** FeedTable.handleToggleRead already implements this correctly (lines 120-135). No change needed.

### Pattern 3: Response.ok check in mutation handlers (SourceList)

const handleDelete = async (id: number) => {

### Pattern 4: TypeScript narrowing for rss-parser custom fields
**What:** Use type assertion with a documented interface instead of `as any` for rss-parser's non-standard fields.
**Example:**
```typescript
// thumbnail.ts
interface RssItem {
  enclosure?: { url?: string };
  'media:content'?: { $?: { url?: string } } | Array<{ $?: { url?: string } }>;
  'media:thumbnail'?: { $?: { url?: string } } | Array<{ $?: { url?: string } }>;
  'itunes:image'?: { $?: { href?: string } };
}

export function extractThumbnailFromItem(item: RssItem): string | null {
  // ... existing logic, now type-safe
}
```

### Pattern 5: optimizePackageImports for lucide-react
**What:** Add `experimental.optimizePackageImports` to `next.config.ts` to tree-shake lucide-react at build time without changing import syntax.
**When to use:** Any Next.js 13.5+ project importing from lucide-react, @radix-ui, or similar barrel-heavy libraries.
**Example:**
```typescript
// next.config.ts
const nextConfig: NextConfig = {
  experimental: {
    optimizePackageImports: ['lucide-react'],
  },
  images: { ... },
};
```
Source: react-best-practices skill, rule `bundle-barrel-imports.md` (HIGH confidence -- official Next.js feature)

### Anti-Patterns to Avoid
- **`as any` for API body fields:** Always use Zod `safeParse` to validate; don't cast unknown JSON
- **Defining components inside components:** Causes remount on every render (skill rule: `rerender-no-inline-components`). No current violations found in codebase.
- **`useEffect` for derived state:** Derive during render instead (skill rule: `rerender-derived-state-no-effect`). Current `useEffect` usages in FeedTable and ChatPanel are legitimate side effects (fetch, IntersectionObserver, scroll sync) -- not candidates for this fix.

## Common Pitfalls

### Pitfall 1: Scope creep during "while I'm here" refactors
**What goes wrong:** A type fix in `thumbnail.ts` prompts adding new URL validation logic, which is D-05 (excluded).
**Why it happens:** Type and validation concerns are adjacent.
**How to avoid:** Image URL validation is excluded by D-05. The `RssItem` interface fix only improves static types; do not add runtime URL validation to `thumbnail.ts` in this phase.
**Warning signs:** Any change that adds a new function or alters return values.

### Pitfall 2: Breaking the `rss-parser` custom field access
**What goes wrong:** Adding types to `rss.ts` line 61 breaks the fallback chain for `content:encoded`.
**Why it happens:** `rss-parser`'s `Item` type doesn't include custom namespace fields.
**How to avoid:** Extend the `CustomItem` type (already used in rss.ts) rather than casting the whole item. Verify `pnpm test` passes after change.
**Warning signs:** TypeScript error `Property 'content:encoded' does not exist`.

### Pitfall 3: Zod parse vs safeParse
**What goes wrong:** Using `.parse()` throws uncaught errors; using `.safeParse()` requires handling the `success: false` branch.
**How to avoid:** Always use `safeParse` in route handlers; return 400 on failure. Never use `.parse()` in a route handler without a surrounding try-catch.

### Pitfall 4: optimizePackageImports in `experimental` key
**What goes wrong:** Adding to wrong config key (e.g. root level).
**How to avoid:** It lives under `experimental: { optimizePackageImports: [...] }` in `next.config.ts`. No import changes needed anywhere -- existing `import { X } from 'lucide-react'` statements remain valid.

  isRead: z.boolean(),
});

export async function PATCH(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  const articleId = parseInt(id, 10);
  if (isNaN(articleId)) {
