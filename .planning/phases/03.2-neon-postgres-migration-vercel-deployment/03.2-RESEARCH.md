## Summary

This phase migrates the project from local SQLite (better-sqlite3) to Neon Postgres via Vercel Marketplace, then deploys to Vercel with working cron jobs. The migration is entirely mechanical: swap the Prisma provider, add the `directUrl` for migrations, update the `topics` field type, and run data migration for articles from March 18th onward.

The most significant code-wide change is the `topics` field type upgrade from `String?` to `Json?`. With Postgres, Prisma returns `topics` as a parsed `Prisma.JsonValue` (an actual array), not a JSON string. This breaks `briefing.ts`'s `parseTopics(raw: string | null)` function, the `enrich.ts` write path that calls `JSON.stringify(result.topics)`, the `ArticleRow` type, and all related tests. All of these must be updated as part of the schema migration task.

The Vercel-Neon integration injects exactly `DATABASE_URL` (pooled) and `DATABASE_URL_UNPOOLED` (direct). The decision D-07 to use `directUrl = env("DATABASE_URL_UNPOOLED")` is confirmed correct. The project's build script already runs `prisma migrate deploy`, so adding a `postinstall` script for `prisma generate` is the main Vercel deployment concern.

**Primary recommendation:** Execute in this order: (1) provision Neon + pull env vars, (2) swap schema + delete SQLite migrations + run fresh migrate dev, (3) update `topics` type downstream, (4) run data migration script, (5) clean up SQLite deps, (6) update cron schedule, (7) deploy.

## Architecture Patterns

### Pattern 1: Neon + Prisma 5 Schema Configuration

**What:** Add `provider = "postgresql"` and `directUrl` for migrations; `DATABASE_URL` is pooled (PgBouncer), `DATABASE_URL_UNPOOLED` is direct.

**Source:** Neon docs (https://neon.com/docs/guides/prisma) + confirmed by Vercel-Neon integration variable names.

```prisma
// prisma/schema.prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DATABASE_URL_UNPOOLED")
}
```

`DATABASE_URL` routes through Neon's PgBouncer pooler (hostname has `-pooler` in it). Used by the running app. `DATABASE_URL_UNPOOLED` is the direct connection used by `prisma migrate dev/deploy`.

### Pattern 2: Prisma Client Singleton for Neon (no adapter)

**What:** The existing singleton in `src/lib/prisma.ts` works as-is. No adapter import needed.

```typescript
// src/lib/prisma.ts -- no changes needed for connection logic
import "server-only";
import { PrismaClient } from "@prisma/client";

const globalForPrisma = global as unknown as { prisma: PrismaClient };

export const prisma = globalForPrisma.prisma ?? new PrismaClient();

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

**Note:** Vercel cold-start creates a new instance per function invocation anyway. The global singleton prevents multiple instances during dev hot-reload only.

### Pattern 3: Vercel Build + postinstall for Prisma Generate

**What:** Vercel caches `node_modules` between deployments. Without a `postinstall` script, `prisma generate` only runs on initial deploy, causing "Outdated Prisma Client" errors when schema changes.

**Source:** https://www.prisma.io/docs/orm/prisma-client/deployment/serverless/deploy-to-vercel

The existing build script is:
```json
"build": "prisma generate && prisma migrate deploy && next build"
```

This already covers it via the `build` script. However, adding `postinstall` is still recommended as a safety net for Vercel's caching behavior:

```json
"postinstall": "prisma generate"
```

**Decision (Claude's discretion):** Add `postinstall`. It is the standard Vercel + Prisma pattern. It is idempotent and harmless.

### Pattern 4: Fresh Migration History (SQLite -> Postgres)

**What:** SQLite and Postgres migration SQL is incompatible. Delete the old migration directory, create a single fresh `init` migration against the Postgres schema.

```bash
# After schema.prisma provider is updated to postgresql and DATABASE_URL points to Neon:
rm -rf prisma/migrations
npx prisma migrate dev --name init
```

This creates a clean `prisma/migrations/TIMESTAMP_init/migration.sql` with Postgres-compatible DDL.

**Source:** Prisma docs on handling provider mismatches (https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/limitations-and-known-issues)

### Pattern 5: Dual-Client Data Migration Script

**What:** A standalone script that creates two Prisma clients -- one pointing at `./prisma/dev.db` (SQLite, read-only source), one pointing at Neon via `DATABASE_URL` (Postgres, write target).

**Decision (Claude's discretion):** Iterate-and-upsert pattern. With only ~366 articles, batching is unnecessary complexity. Single `createMany` with `skipDuplicates` (Postgres supports it) per entity type is the cleanest approach.

```typescript
// scripts/migrate-data.ts (rough structure)
import { PrismaClient as PrismaClientSqlite } from '@prisma/client';
import { PrismaClient as PrismaClientPg } from '@prisma/client';

const sqlite = new PrismaClientSqlite({
  datasources: { db: { url: 'file:./prisma/dev.db' } },
});

const pg = new PrismaClientPg({
  datasources: { db: { url: process.env.DATABASE_URL! } },
});
```

**Important:** Both clients use the same generated `@prisma/client`. After the schema switch to Postgres, the generated client is Postgres-typed. To read SQLite you pass a `datasources` override with `file:` protocol. Prisma 5 supports this -- the override bypasses the schema's `provider` for connection purposes but the SQL issued still needs to match the engine. This means the SQLite client override ONLY works while the generated client is still SQLite-compatible (before schema switch) OR you use a separate pre-switch snapshot.

**Revised approach:** Run the migration script BEFORE switching the schema to Postgres, while the generated client is still SQLite-compatible. Steps:
1. Write and commit `scripts/migrate-data.ts` (while schema is still SQLite)
2. Run the script locally to migrate data to Neon (using env var override for pg URL)
3. Then switch schema to Postgres and delete SQLite migrations

Actually, there is a simpler approach that avoids the sequencing complexity: the datasource override in `new PrismaClient({ datasources: { db: { url: 'file:...' } } })` only overrides the connection URL. The schema provider type still determines what SQL is generated. For reading SQLite data you don't need any Prisma-specific SQL features; simple `findMany` works identically in both providers. So after switching to Postgres schema:

- The `sqlite` client with `datasources` override of `file:./prisma/dev.db` will attempt to open the SQLite file with the Postgres client binary -- this WILL FAIL. The Prisma query engine binary is provider-specific.

**Correct approach for dual-client script:** Use the `tsx` runtime with two separate Prisma schema files, OR keep schema as SQLite during data migration step, OR use a raw SQL approach.

**Recommended implementation (Claude's discretion):** Keep the schema as SQLite, point `DATABASE_URL` at the SQLite file, run the migration script against Neon using the unpooled URL as `DIRECT_URL`, then switch the schema. Specifically:

```typescript
// scripts/migrate-data.ts
// Run BEFORE prisma provider swap, while schema is still "sqlite"
// DATABASE_URL = file:./prisma/dev.db (SQLite, for reading)
// NEON_URL = postgres://... (Neon, for writing -- passed separately)

const sqliteClient = new PrismaClient(); // reads dev.db via DATABASE_URL
const pgClient = new PrismaClient({
  datasources: { db: { url: process.env.NEON_URL! } },
});
// pgClient uses Postgres wire protocol directly via the Neon URL
// This STILL fails because the query engine binary is SQLite-specific
```

**The cleanest approach after research:** Use `better-sqlite3` directly to read the SQLite file as raw JSON, then use the Postgres Prisma client (after schema switch) to write. This avoids dual-engine issues entirely.

```typescript
// scripts/migrate-data.ts (recommended pattern)
// Run AFTER schema switch (Postgres client is active)
// Reads dev.db using better-sqlite3 directly (raw JS object reads, no Prisma engine)
// Writes to Neon via the standard prisma client

import Database from 'better-sqlite3';
import { PrismaClient } from '@prisma/client';

const db = new Database('./prisma/dev.db', { readonly: true });
const prisma = new PrismaClient(); // connects to Neon via DATABASE_URL from .env.local

const CUTOFF = new Date('2026-03-18T00:00:00.000Z');
```

This approach:
- Is provider-agnostic (reads SQLite as raw JS objects via `better-sqlite3`)
- Runs after the schema switch so the Prisma client is Postgres-compatible
- Requires `better-sqlite3` to still be in `dependencies` when the script runs
- `better-sqlite3` removal (D-13) happens AFTER the migration script is deleted from the codebase

**Note on `topics` in migration script:** Since `topics` in SQLite is stored as a JSON string (e.g., `'["model releases"]'`), and the Postgres schema uses `Json?`, the script must parse the string before writing: `JSON.parse(row.topics ?? 'null')`.

### Pattern 6: topics Field Type Change -- Full Impact Map

When `topics` changes from `String?` to `Json?` in the Prisma schema, the generated TypeScript type changes from `string | null` to `Prisma.JsonValue | null`. `Prisma.JsonValue` is `string | number | boolean | null | JsonObject | JsonArray`. For topics specifically, it will always be a `string[]` (array of strings) at runtime.

**Files requiring updates:**

| File | Current behavior | Required change |
|------|-----------------|-----------------|
| `src/lib/enrich.ts` L107 | `topics: JSON.stringify(result.topics)` | `topics: result.topics` (pass array directly) |
| `src/lib/briefing.ts` L16 | `parseTopics(raw: string \| null)` | Update signature to `parseTopics(raw: Prisma.JsonValue \| null)`, remove `JSON.parse` |
| `src/types/index.ts` L12 | `topics: string \| null` | `topics: Prisma.JsonValue \| null` (import `Prisma` from `@prisma/client`) |
| `src/lib/enrich.test.ts` L102,118 | expects `JSON.stringify` call | update to expect plain array |
| `src/lib/briefing.test.ts` L44+ | fixture strings like `'["AI safety"]'` | update to arrays `['AI safety']` |

**Simplified `parseTopics` after Json type:**
```typescript
// src/lib/briefing.ts (after topics: Json?)
import { Prisma } from '@prisma/client';

export function parseTopics(raw: Prisma.JsonValue | null): string[] {
  if (!raw) return ['Uncategorized'];
  if (Array.isArray(raw) && raw.length > 0) return raw as string[];
  return ['Uncategorized'];
}
```

No `JSON.parse` needed -- Prisma already returns the parsed value.

### Pattern 7: mode: 'insensitive' for Contains Filters

In `src/app/api/articles/route.ts`, the `contains` filters need `mode: 'insensitive'` now that the database is Postgres (SQLite silently ignores this option, Postgres requires explicit case-sensitivity handling).

```typescript
// src/app/api/articles/route.ts (after migration)
if (search) {
  andConditions.push({
    OR: [
      { title: { contains: search, mode: 'insensitive' } },
      { source: { name: { contains: search, mode: 'insensitive' } } },
    ],
  });
}
```

### Anti-Patterns to Avoid

- **Using `@neondatabase/serverless` adapter:** D-05 locked this out. Standard `PrismaClient` over TCP pooling is sufficient for this scale.
- **Running `prisma migrate dev` on Vercel:** Always use `prisma migrate deploy` in production. `migrate dev` is for local development only and creates shadow databases.
- **Switching provider before the data migration:** The migration script reads the SQLite file directly via `better-sqlite3`. The SQLite file must still exist when the script runs (it's only deleted after the script completes).
- **Deleting `better-sqlite3` before running the migration script:** Keep it in `dependencies` until `scripts/migrate-data.ts` has been executed and deleted.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Case-insensitive search | Custom SQL `ILIKE` wrapper | `mode: 'insensitive'` in Prisma `contains` | Prisma generates correct Postgres `ILIKE` |

## Common Pitfalls

### Pitfall 1: topics Field -- Double-Parsing on Read

**What goes wrong:** After `topics` becomes `Json?`, calling `JSON.parse(article.topics)` will throw or produce garbage because `article.topics` is already a parsed array, not a string.
**Why it happens:** Developers forget Prisma deserializes `Json` fields automatically.
**How to avoid:** Update `parseTopics` to check `Array.isArray(raw)` directly. Remove all `JSON.parse` calls on topics.
**Warning signs:** Tests pass with string fixtures but runtime throws `SyntaxError: Unexpected token '[' in JSON`

### Pitfall 2: topics Field -- Double-Stringifying on Write

**What goes wrong:** `enrich.ts` currently does `topics: JSON.stringify(result.topics)`. After the type change, this writes a string `'["model releases"]'` to a Json column. Postgres accepts it (stores as a JSON string), but reads back as a string, not an array -- causing parse breakage downstream.
**Why it happens:** The SQLite workaround (store JSON as string) is left in place after the type change.
**How to avoid:** Remove `JSON.stringify` wrapper in `saveEnrichmentResults`. Pass the array directly: `topics: result.topics`.
**Warning signs:** `parseTopics` returns `['Uncategorized']` for all articles even though they were enriched.

### Pitfall 3: Vercel Build Cache -- Stale Prisma Client

**What goes wrong:** Vercel caches `node_modules` between deployments. If the Prisma schema changes, the cached generated client is stale and queries fail at runtime.
**Why it happens:** Prisma's `postinstall` hook doesn't run on cached installs.
**How to avoid:** Add `"postinstall": "prisma generate"` to `package.json` scripts. The existing `build` script already runs `prisma generate` as its first step, which also covers this -- but `postinstall` is the belt-and-suspenders standard.
**Warning signs:** Deployment succeeds but requests throw "PrismaClientKnownRequestError: Unknown field" or similar.

### Pitfall 4: migrate deploy vs migrate dev on Vercel

**What goes wrong:** `prisma migrate dev` creates shadow databases and runs interactively -- it hangs or errors in Vercel's build environment.
**Why it happens:** Developers run the wrong command.
**How to avoid:** Keep `prisma migrate deploy` in the build script (already correct in this project). `migrate dev` is only for local development.

### Pitfall 5: Conflicting Env Vars When Connecting Neon

**What goes wrong:** Vercel shows "Failed to set environment variables" during Neon integration setup.
**Why it happens:** Existing `DATABASE_URL` or `PGHOST` env vars conflict with what Neon tries to inject.
**How to avoid:** Before connecting Neon: go to Vercel > Project Settings > Environment Variables and remove any existing `DATABASE_URL`, `PGHOST`, `PGUSER`, `PGDATABASE`, or `PGPASSWORD` variables first.

### Pitfall 6: SQLite File Read in Migration Script After Provider Swap

**What goes wrong:** Migration script fails with `Error: Cannot find module 'better-sqlite3'` if package is removed before script execution.
**Why it happens:** Cleanup (D-13) is done out of order.
**How to avoid:** Order matters: run and verify migration script, then delete it from git, then remove SQLite dependencies.

### Migration script skeleton (scripts/migrate-data.ts)
```typescript
// Source: D-08, D-09, D-10, D-12 -- Claude's discretion on implementation
// Run AFTER schema switch to Postgres, BEFORE removing better-sqlite3
// Requires: DATABASE_URL in .env.local pointing to Neon (via vercel env pull)

import Database from 'better-sqlite3';
import { PrismaClient } from '@prisma/client';

const CUTOFF = new Date('2026-03-18T00:00:00.000Z');

const sqliteDb = new Database('./prisma/dev.db', { readonly: true });
const prisma = new PrismaClient(); // uses DATABASE_URL from .env.local (Neon)

async function main() {
  // 1. Migrate sources (all of them, idempotent by url)
  const sources = sqliteDb.prepare('SELECT * FROM Source').all() as Array<{
    id: number; name: string; url: string; category: string | null; createdAt: string;
  }>;

  for (const s of sources) {
    await prisma.source.upsert({
      where: { url: s.url },
      create: { name: s.name, url: s.url, category: s.category },
      update: {},
    });
    console.log(`  source: ${s.name}`);
  }

  // 2. Migrate articles from March 18th onward, idempotent by guid
  const articles = sqliteDb.prepare(
    'SELECT * FROM Article WHERE publishedAt >= ? ORDER BY publishedAt ASC'
  ).all(CUTOFF.toISOString()) as Array<{
    guid: string; title: string; link: string; description: string | null;
    thumbnail: string | null; publishedAt: string | null; readTimeMin: number | null;
    isRead: number; summary: string | null; topics: string | null;
    importanceScore: number | null; enrichedAt: string | null;
    createdAt: string; sourceId: number;
  }>;

  let created = 0; let skipped = 0;
  for (const a of articles) {
    const existing = await prisma.article.findUnique({ where: { guid: a.guid } });
    if (existing) { skipped++; continue; }

    // Find the new sourceId by matching url
    const srcRow = sqliteDb.prepare('SELECT url FROM Source WHERE id = ?').get(a.sourceId) as { url: string };
    const pgSource = await prisma.source.findUnique({ where: { url: srcRow.url } });
    if (!pgSource) { console.warn(`  skip article (source not found): ${a.guid}`); continue; }

    await prisma.article.create({
      data: {
        guid: a.guid, title: a.title, link: a.link,
        description: a.description, thumbnail: a.thumbnail,
        publishedAt: a.publishedAt ? new Date(a.publishedAt) : null,
        readTimeMin: a.readTimeMin,
        isRead: Boolean(a.isRead),
        summary: a.summary,
        topics: a.topics ? JSON.parse(a.topics) : null,  // String -> Json
        importanceScore: a.importanceScore,
        enrichedAt: a.enrichedAt ? new Date(a.enrichedAt) : null,
        sourceId: pgSource.id,
      },
    });
    created++;
  }

  console.log(`Done. ${created} articles created, ${skipped} skipped.`);
}

main().catch(console.error).finally(() => {
  sqliteDb.close();
  prisma.$disconnect();
});
```

## Open Questions

1. **Source ID remapping in migration script**
   - What we know: Source IDs in SQLite are autoincrement. After seeding Postgres, source IDs may differ (depends on insertion order and whether any IDs were deleted in SQLite).
   - What's unclear: Whether IDs will be 1:1 between SQLite and Postgres after seed.
   - Recommendation: Always look up source by `url` in Postgres (not by `id`) when inserting articles. The migration script skeleton above does this correctly.

2. **Current vercel.json schedule vs D-14**
   - What we know: Current `vercel.json` has `0 4 * * *` and `0 5 * * *`. D-14 specifies 05:00 and 06:00 UTC.
   - What's unclear: Whether the 1-hour shift matters to the user.
   - Recommendation: Update to 05:00/06:00 per D-14 during the vercel.json cron update task.

3. **CRON_SECRET in Vercel environment**
   - What we know: `CRON_SECRET` must be set in Vercel env vars for cron auth to work (existing pattern from CLAUDE.md).
   - What's unclear: Whether it is already set in the Vercel project, or needs to be added during this phase.
   - Recommendation: Include a step to verify/add `CRON_SECRET` to Vercel environment variables as part of deployment.
