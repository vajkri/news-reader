## Summary

This phase corrects a mislabeled data entry: the existing "TLDR Tech" source record currently points to Hacker News (`hnrss.org/frontpage`) instead of the actual TLDR newsletter feed. The fix involves updating that URL to the tldr-rss middleman service, adding a correctly-named Hacker News source so HN articles keep flowing, and adding two new TLDR sources (AI, Design).

The tldr-rss public instance at `https://bullrich.dev/tldr-rss/` is verified live and serving valid RSS 2.0 feeds for tech, ai, and design categories. Each feed produces individual items with proper guid, pubDate, title, link, and description fields -- fully compatible with the existing `fetchFeed()` parser in `src/lib/rss.ts`.

The critical technical constraint is the Prisma `@unique` on `Source.url`. Seed's `upsert` is keyed on `url`, meaning a URL change on an existing record cannot happen via upsert alone: the old URL record won't match, and a new record is created instead while the old record remains. The plan must address this explicitly -- either via a Prisma migration script that runs `UPDATE` directly, or via a two-step seed that deletes the stale record. This is the only real technical decision left to the planner (per Claude's Discretion).

**Primary recommendation:** Use a direct Prisma `update` in a one-off migration script (not seed.ts alone) to swap the URL on the existing record, then add the new sources via seed upsert. This preserves the existing article-to-source relationship for already-fetched articles.

## Architecture Patterns

### The URL-Keyed Upsert Constraint

The existing `prisma/seed.ts` uses:

```typescript
await prisma.source.upsert({
  where: { url: source.url },
  create: source,
  update: {},
});
```

The `update: {}` means even if a record is found by URL, nothing is updated on it. More critically, the lookup key IS the URL. Changing a source's URL from `A` to `B` in the SOURCES array means:

- Old record with URL `A` is never touched (no delete, no update)
- A new record with URL `B` is created
- Result: two Source records exist, one stale

This is the core issue the planner must solve.

### Recommended Approach: Explicit Update + Upsert Seed

```typescript
// Step 1: Update the existing TLDR Tech URL in place (preserves source.id and article relations)
await prisma.source.update({
  where: { url: "https://hnrss.org/frontpage" },  // old URL
  data: { url: "https://bullrich.dev/tldr-rss/tech.rss" },
});

// Step 2: Seed upserts handle the rest
// - "TLDR Tech" now matches by new URL (already exists, update:{} is fine)
// - "Hacker News" is new, gets created
// - "TLDR AI" is new, gets created
// - "TLDR Design" is new, gets created
```

This can be either:
- A standalone migration script (`prisma/migrate-sources.ts`), run once
- Embedded as a preamble in `seed.ts` (less clean but fewer files)

The planner chooses based on Claude's Discretion.

### Recommended Project Structure Changes

```
prisma/
├── seed.ts              # Update SOURCES array; add HN, TLDR AI, TLDR Design
```

### Anti-Patterns to Avoid

- **Relying on seed.ts upsert alone to change a URL:** The upsert `where: { url }` means the old URL record is never touched; a new record is created instead, leaving a stale orphan source in the DB.
- **Deleting the TLDR Tech source and recreating it:** This would orphan all existing articles (sourceId FK would become invalid or cascade-delete articles per `onDelete: Cascade` in the schema). Preserve the existing record.
- **Changing `update: {}` to patch name/url in seed:** Seed is designed to be idempotent and run repeatedly; mutating URL in update makes reruns unpredictable.

## Common Pitfalls

### Pitfall 1: Orphan Source Record After URL Change

**What goes wrong:** Developer updates TLDR Tech URL in SOURCES array and runs `npm run seed`. The upsert finds no record matching the new URL, creates a new Source. The old "TLDR Tech" record (pointing to HN) remains in DB. Feed cron now fetches both HN (old record) and TLDR tech (new record).

**Why it happens:** `upsert where: { url }` uses the new URL as the lookup key; old record never matched.

**How to avoid:** Run an explicit `prisma.source.update({ where: { url: OLD_URL }, data: { url: NEW_URL } })` before the seed upsert. Either in a migration script or as a preamble in seed.ts guarded by existence check.

**Warning signs:** After migration, `prisma.source.findMany()` returns 5 sources instead of the expected 4 sources pre-migration / 5 sources post-migration (TLDR Tech, TLDR AI, TLDR Design, Hacker News, Dev.to, Smashing Magazine = 6). Wait, current count is 4; target is 6 (TLDR Tech + TLDR AI + TLDR Design + Hacker News + Dev.to + Smashing Magazine). If you see 7, a stale record was left.

### Pitfall 2: Cascade Delete Destroys Existing Articles

**What goes wrong:** Developer deletes the existing "TLDR Tech" source (to avoid URL conflict) and re-creates it. The schema has `onDelete: Cascade` on articles, so all articles under that sourceId are deleted.

**Why it happens:** Prisma cascade delete on Article.sourceId.

**How to avoid:** Never delete and recreate -- always update the URL on the existing record in place.

**Warning signs:** Article count drops significantly after migration.

### Pitfall 3: TLDR Feeds Return Only Today's Articles

**What goes wrong:** The tldr-rss feeds have a rolling window; they only return the current day's digest items (published "today"). The `fetchFeed()` function has a `maxAgeMs` default of 24 hours, so items older than that are filtered. This is expected behavior for TLDR -- no backfill of historical articles.

**Why it happens:** TLDR publishes one digest per day; tldr-rss splits that digest into items, all with the same pubDate.

**How to avoid:** Understand this is working as intended. First fetch will import ~12 items from today's TLDR edition.

**Warning signs:** TLDR source shows 0 articles imported -- check if the daily digest was published (may not exist on weekends or holidays).

### Pitfall 4: Duplicate guids Between TLDR Tech (Old HN) and New Hacker News Source

**What goes wrong:** The new "Hacker News" source (hnrss.org/frontpage) has the same feed URL as the old "TLDR Tech" source. Articles fetched via HN url in the past have guids like `https://news.ycombinator.com/item?id=...`. The new Hacker News source will try to re-fetch these, but guid deduplication in `fetch/route.ts` will block re-insertion.

**Why it happens:** The fetch route deduplicates by `guid` globally (not per-source). Same guid = no re-import.

**How to avoid:** This is actually correct behavior. Pre-existing HN articles stay attributed to "TLDR Tech" (as decided). New HN fetches are silently skipped for already-known guids, and only NEW HN articles get imported under the new "Hacker News" source. No action needed.

## Code Examples

### Pattern: One-off URL swap (migration script)

```typescript
// Source: established Prisma pattern, prisma/migrate-sources.ts
import { PrismaClient } from "@prisma/client";
const prisma = new PrismaClient();

async function main() {
  // Swap the URL on the existing TLDR Tech record (preserves id + article relations)
  const existing = await prisma.source.findUnique({
    where: { url: "https://hnrss.org/frontpage" },
  });
  if (existing && existing.name === "TLDR Tech") {
    await prisma.source.update({
      where: { url: "https://hnrss.org/frontpage" },
      data: { url: "https://bullrich.dev/tldr-rss/tech.rss" },
    });
    console.log("Updated TLDR Tech URL");
  } else {
    console.log("TLDR Tech not found at old URL -- skipping (already migrated?)");
  }
}

main().catch(console.error).finally(() => prisma.$disconnect());
```
