## Summary

This phase redesigns the enrichment pipeline from a fragile daily cron to a multi-trigger system, and replaces the date-window briefing model with a watermark-based "catch me up" view. The existing code is well-structured: `enrich.ts` already has a time-budget loop; the changes are additive (increase BATCH_LIMIT, flip orderBy, add schema fields, add GitHub Actions workflow, add client feedback components).

The largest new surface areas are: (1) the watermark model (new DB field or cookie, new queries), (2) the feedback table (new Prisma model, new API route), (3) the calibration prompt injection (pre-enrichment query + string assembly), and (4) the on-demand "Enrich now" button (client component + fetch call). All four are well within existing project patterns. No new third-party libraries are required.

The 48h staleness cutoff and the archive "top 15 by importance" freeze require careful Prisma query design. The feedback pattern extraction is pure string logic over DB records -- no ML needed.

**Primary recommendation:** Build in six sequential plans: (1) schema migrations, (2) enrichment pipeline fixes, (3) GitHub Actions workflow, (4) watermark briefing model, (5) on-demand enrich + pending section, (6) feedback UI + calibration injection.

### Locked Decisions

- **D-01:** Default briefing mode is "catch me up": shows all enriched articles since the user's last-read watermark, not a fixed daily window

- **D-05:** Archive mode via DateStepper shows frozen top 15 articles by importance for that date, regardless of how many enrichment runs happened. Archive has no feedback buttons or "Enrich now" button. Clear "Archive" banner distinguishes it from live view.
- **D-06:** GitHub Actions workflow replaces Vercel cron jobs. Single workflow file runs every 4 hours, calling /api/fetch then /api/enrich sequentially. CRON_SECRET stored as GitHub repo secret.
- **D-07:** Remove both cron entries from vercel.json entirely
- **D-08:** On-demand enrichment on briefing page: when unenriched articles exist, show them in an amber "pending enrichment" section with an "Enrich now (N)" button. User clicks to trigger, page reloads with results. No auto-enrichment on page load.

- **D-10:** Keep maxDuration = 60 on the enrich route. The existing time-budget loop handles multiple iterations within the budget.
- **D-11:** Enrichment scope: only articles fetched recently. If an article wasn't enriched within 48h of being fetched, it's abandoned as stale.
- **D-12:** fetchUnenrichedArticles changes from publishedAt ASC (oldest first) to publishedAt DESC (newest first)

- **D-14:** Each article in the "catch me up" view has up/down arrow buttons for importance feedback. Feedback controls only appear on current articles, not archive.

- **D-16:** Upvote reasons: "Directly useful for my work", "Important industry shift", "Great summary quality", "Would have missed this"
- **D-17:** Feedback closes the loop via prompt-based calibration. Before each enrichment run, query accumulated feedback, extract patterns, inject calibration section into system prompt. No ML.
- **D-18:** Feedback data stored as: articleId, direction (up/down), reasons (array), plus article's source, topics, contentType for pattern extraction.

### Pattern 1: Watermark Storage -- DB Singleton Row

**Recommendation:** Store watermark as a single `UserPreference` row in Postgres (singleton for single-user app). This avoids cookie size limits, works across devices, and is queryable server-side without JS.

**Schema:**
```prisma
model UserPreference {
  id              Int      @id @default(autoincrement())
  key             String   @unique
  value           String
  updatedAt       DateTime @updatedAt
}
```

Use `key = 'briefing_watermark'`, `value = ISO timestamp string`. On briefing page visit (server component), read the watermark, render page, then trigger a server action to update it.

**Why not cookie:** Cookies require client component or middleware to set; server components can read but not write cookies directly in Next.js App Router without `cookies().set()` inside a route handler or server action. A DB singleton is simpler and queryable.

**Why not localStorage:** Not accessible server-side, defeats Server Component data fetching pattern.

**Confidence:** MEDIUM -- singleton DB row is the cleanest pattern for a single-user app with no auth. Requires a new Prisma model and migration.

### Pattern 2: Watermark Update via Server Action

```typescript
// src/lib/watermark.ts
'use server';
import { prisma } from './prisma';

export async function getWatermark(): Promise<Date | null> {
  const pref = await prisma.userPreference.findUnique({
    where: { key: 'briefing_watermark' },
  });
  return pref ? new Date(pref.value) : null;
}

export async function updateWatermark(ts: Date): Promise<void> {
  await prisma.userPreference.upsert({
    where: { key: 'briefing_watermark' },
    update: { value: ts.toISOString() },
    create: { key: 'briefing_watermark', value: ts.toISOString() },
  });
}
```

Briefing page (Server Component) calls `updateWatermark(new Date())` after reading the watermark for the current render. This records "user visited at T" so next visit shows articles enriched after T.

### Pattern 9: GitHub Actions Workflow

```yaml
# .github/workflows/enrich.yml
name: Fetch and Enrich

on:
  schedule:
    - cron: '0 */4 * * *'  # Every 4 hours
  workflow_dispatch:         # Manual trigger

jobs:
  fetch-and-enrich:
    runs-on: ubuntu-latest
    steps:
      - name: Fetch articles
        run: |
          curl -f -s \
            -H "Authorization: Bearer ${{ secrets.CRON_SECRET }}" \
            ${{ vars.APP_URL }}/api/fetch

      - name: Enrich articles
        run: |
          curl -f -s \
            -H "Authorization: Bearer ${{ secrets.CRON_SECRET }}" \
            ${{ vars.APP_URL }}/api/enrich
```

Use `vars.APP_URL` (GitHub Actions variable, not secret) for the production URL. `secrets.CRON_SECRET` matches the existing `CRON_SECRET` env var on Vercel.

**Key point:** GitHub Actions `schedule` uses UTC cron. Every 4 hours = `0 */4 * * *`.

### Pattern 10: On-Demand EnrichNow Button

```tsx
}
```

### Anti-Patterns to Avoid

- **Using `enrichedAt` for archive day queries:** Archive must use `publishedAt` window (D-13). Using `enrichedAt` would break if articles are re-enriched.
- **Auto-enriching on page load:** D-08 is explicit -- no auto-enrichment. User-initiated only.
- **Updating watermark before reading it:** Read watermark first to determine new/reviewed split; then update watermark to `now()` so next visit is correct.
- **Storing feedback reasons as a flat string:** Use `Json` (JSONB in Postgres) for `reasons` array -- matches the `topics` field pattern already in the schema.
- **Putting calibration logic in the briefing page:** Calibration belongs in `enrich.ts` -- it runs before enrichment, not at display time.

## Common Pitfalls

### Pitfall 1: Watermark Race Condition on First Visit

**What goes wrong:** First-ever visit has no watermark. All enriched articles appear as "new". If watermark is written _after_ the query, a second load within milliseconds also sees everything as new.

**Why it happens:** Watermark is read and written in the same server render.

**How to avoid:** Read watermark, snapshot it for the query, then write `new Date()` immediately. Use the snapshotted value for queries -- not the just-written value.

**Warning signs:** Every page reload shows all articles as new.

### Pitfall 2: `createdAt` vs `publishedAt` for Staleness

**What goes wrong:** Using `publishedAt` for the 48h staleness cutoff catches articles that were published a long time ago but just fetched (e.g., sitemap sources sometimes surface older articles).

**Why it happens:** `publishedAt` is the article's original publication date; `createdAt` is when the row was inserted into our DB (i.e., when it was fetched).

**How to avoid:** Use `createdAt >= cutoff` for the staleness filter in `fetchUnenrichedArticles`. The Article schema already has `createdAt DateTime @default(now())` -- no migration needed.

**Warning signs:** Articles that are months old appearing in pending section; enrichment skipping recently-fetched articles because their publishedAt is old.

### Pitfall 3: BATCH_LIMIT Test Hardcoded to 15

**What goes wrong:** `enrich.test.ts` has `expect(BATCH_LIMIT).toBe(15)`. After changing to 25, the test fails.

**How to avoid:** Update the test assertion to `toBe(25)` when changing the constant.

### Pitfall 4: Archive "Top 15" vs Catch-me-up "All"

**What goes wrong:** Archive mode accidentally renders all articles instead of top 15; or live mode accidentally caps at 15.

**How to avoid:** Archive path: always `take: 15`. Catch-me-up path: no `take` limit (show all new articles). Gate on `dateParam` presence to switch modes.

### Pitfall 5: GitHub Actions `curl -f` Exits on HTTP Errors

**What goes wrong:** If `/api/fetch` returns a non-2xx (e.g., 401, 500), `curl -f` exits the step with error and blocks `/api/enrich` from running.

**How to avoid:** Use `curl -f` intentionally -- you want the workflow to fail visibly on auth errors. But ensure `APP_URL` env var is set in repo variables and `CRON_SECRET` in repo secrets before the first run.

### Pitfall 6: Feedback Client Component in Server Component Tree

**What goes wrong:** `FeedbackButtons` needs `'use client'` but is rendered inside a Server Component. If it imports anything server-only, it will fail at build.

**How to avoid:** Keep feedback components purely client-side. Pass serialized article data as props (no Prisma types). Use the existing `ArticleRow` interface which is already JSON-safe.

### Pitfall 7: `NEXT_PUBLIC_CRON_SECRET` Exposure

**What goes wrong:** Exposing CRON_SECRET via NEXT_PUBLIC_ makes it visible in browser bundle.

**How to avoid:** Use dedicated `/api/enrich-on-demand` route (no auth, single-user app). Or, if reusing `/api/enrich`, do the fetch server-side via a Server Action instead of client-side fetch.

**Recommendation:** Server Action approach avoids the exposure entirely:
```typescript
// src/lib/actions.ts (already exists)
'use server';
export async function triggerEnrichment() {
  const res = await fetch(`${process.env.NEXT_PUBLIC_APP_URL}/api/enrich`, {
    headers: { Authorization: `Bearer ${process.env.CRON_SECRET}` },
  });
  return res.ok;
}
```

## Open Questions

1. **Watermark on first visit: show all enriched or show today only?**
   - What we know: D-04 says "all caught up" when no new articles, with today's briefing collapsed below
   - What's unclear: On _very first visit_ (null watermark), should everything be "new" or should we seed the watermark to 24h ago?
   - Recommendation: Seed watermark to 24h ago on first visit -- prevents overwhelming new users with the full backlog. Implement as: `watermark ?? new Date(Date.now() - 24 * 60 * 60 * 1000)`

2. **Server Action vs. client fetch for EnrichNow:**
   - Resolved in research: prefer Server Action in `actions.ts` to avoid exposing CRON_SECRET client-side. Server Action calls `/api/enrich` with the secret server-side.
   - Implementation detail left to planner: trigger via `<form action={serverAction}>` or `startTransition(serverAction)`.

3. **Feedback table: allow multiple feedback entries per article or upsert?**
   - Recommendation: Allow multiple (append-only) for pattern analysis -- if user changes their mind and votes up then down, both signals are useful. Simple `create` not upsert.
