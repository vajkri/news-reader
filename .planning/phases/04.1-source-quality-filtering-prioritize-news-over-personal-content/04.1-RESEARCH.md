## Summary

This phase has four sequential work streams: (1) source curation, (2) schema migration, (3) prompt and model improvement, (4) full backfill re-enrichment. Each depends on the previous.

Source curation removes Dev.to and adds 2-3 high-signal AI/ML feeds. The schema migration adds `contentType String?` and `thinContent Boolean?` to the Article model as nullable columns (existing rows remain NULL until re-enrichment). The enrichment pipeline and chat API system prompts get improved using the `chatbot-prompt-design` skill findings. A model evaluation run compares 3-4 candidates against a test batch. The winning model is locked into `ai.ts`, then a one-time re-enrichment script backfills all existing articles with fresh summaries, scores, topics, contentType, and thinContent.

No UI changes are in scope. The stored fields are for future use (Phase 5).

**Primary recommendation:** Execute in order: source seed change + schema migration, then prompt rewrite, then model evaluation, then Dev.to article deletion + full re-enrichment script.

### Pattern 1: Source Seed Deletion + Addition

The existing seed.ts uses an upsert-by-URL pattern. Removing Dev.to requires a `deleteMany` preamble (same pattern as the URL-swap preamble in Phase 03.1). Adding new sources is a straightforward array addition.

**Dev.to deletion preamble in seed.ts:**
```typescript
// One-time: remove Dev.to source and all its articles (cascades via onDelete: Cascade)
await prisma.source.deleteMany({ where: { url: 'https://dev.to/feed' } });
```

The `Article` model has `onDelete: Cascade` on its `Source` relation, so deleting the source row cascades to all Dev.to articles automatically. No separate article deletion step needed in seed.ts.

**However:** The re-enrichment script (D-17) should run AFTER seed changes are applied. The seed only removes the source; the cascade handles articles. Seed is re-runnable (upserts are idempotent); the `deleteMany` preamble is also safe to re-run (no-op if source is already gone).

### Pattern 2: Prisma Schema Column Addition

Adding nullable columns to an existing table. Both new fields should be `String?` and `Boolean?` with no `@default`:

```prisma
model Article {
  // ... existing fields ...
  contentType     String?
  thinContent     Boolean?
}
```

Nullable with no default is correct for Postgres: existing rows will be NULL until backfilled. Avoids the Prisma SQLite migration bug #26571 (which was the reason `topics` has no `@default`). With Postgres this issue does not apply, but consistent pattern is cleaner.

Migration command: `npx prisma migrate dev --name add_content_type_thin_content`

### Pattern 4: Re-enrichment Script

Follows the Phase 03.2 `scripts/migrate-data.ts` precedent. Key differences:

- Reads from Neon Postgres (not SQLite) via the normal `prisma` singleton
- Processes in batches of 15 (matching `BATCH_LIMIT`)
- Resets `enrichedAt = null` approach is NOT used; instead, the script directly calls the enrichment pipeline on all articles (or a subset by sourceId exclusion for Dev.to)
- Reports progress per batch
- Handles partial failures gracefully (log and continue)

```typescript
// scripts/re-enrich.ts sketch
import { prisma } from '../src/lib/prisma';
import { enrichArticlesBatch, saveEnrichmentResults, BATCH_LIMIT } from '../src/lib/enrich';

async function main() {
  let offset = 0;
  let total = 0;
  while (true) {
    const articles = await prisma.article.findMany({
      where: { /* exclude dev.to articles already deleted */ },
      select: { id: true, title: true, description: true, source: { select: { name: true } } },
      orderBy: { publishedAt: 'desc' },
      skip: offset,
      take: BATCH_LIMIT,
    });
    if (articles.length === 0) break;
    const results = await enrichArticlesBatch(articles);
    const { saved, errors } = await saveEnrichmentResults(results);
    total += saved;
    console.log(`Batch ${offset / BATCH_LIMIT + 1}: saved ${saved}, errors: ${errors.length}`);
    offset += BATCH_LIMIT;
  }
  console.log(`Done. Total enriched: ${total}`);
}
```

The script runs with: `npx tsx scripts/re-enrich.ts`

### Pattern 5: Enrichment Prompt Improvement (chatbot-prompt-design skill)

The skill provides a checklist and anti-pattern table. Applied to the enrichment `SYSTEM_PROMPT`:

**Current prompt anti-patterns identified:**
- "Score importance 1-10 relative to the OTHER articles in this batch" -- correct behavior, but the instruction doesn't calibrate what 1 vs 10 means. Risk of score compression (everything 4-7).
- No explicit guidance on contentType assignment logic or the 5-type taxonomy.
- thinContent description is in the Zod schema only, not reinforced in SYSTEM_PROMPT instructions.

**Improved enrichment SYSTEM_PROMPT structure (to be drafted by planner):**
```
## Identity & Scope
- AI analyst tool for an AI-focused frontend developer's news briefing
- Tasks: summarize, classify topic, score importance, classify content type, flag thin content

## Summary Rules
- 2-3 sentences, analyst briefing tone (factual, dry, implications-forward)
- Final sentence MUST start with "This" and state the implication
- Avoid generic phrases ("In a recent post...", "The article discusses...")

## Importance Scoring (1-10)
- Score relative to other articles in the batch
- Anchors: 9-10 = industry-defining (GPT-5 launch, major regulation passed), 7-8 = significant (new model release, major tool update), 4-6 = notable (tool update, research paper), 1-3 = low signal (minor update, personal post)
- Weight: 60% broad AI industry impact + 40% relevance to frontend developer building with AI tools

## Topic Classification
- Prefer seeds: [list]
- Create new category ONLY when no seed fits AND the topic is recurring
- Assign 1-3 topics; most articles need only 1

## Content Type (ONE per article)
- news: factual reporting of an event or release
- tutorial: how-to, guide, step-by-step instructions
- opinion: analysis, commentary, editorial viewpoint
- announcement: official company/org announcement
- roundup: weekly/monthly digest, list of things

## Thin Content
- Set thinContent=true if RSS description is < 50 words OR consists only of generic boilerplate with no article-specific content
```

### Pattern 6: Chat SYSTEM_PROMPT Improvement

Current prompt assessed against the chatbot-prompt-design skill checklist:

**Gaps found:**
- No named welcome/intro capability statement (violates "name specific capabilities")
- "I'm here to help with anything about AI news" risk -- current prompt says "You answer questions about AI news articles" which is slightly vague
- No explicit "answer-first" rule
- No "fail fast" instruction -- instead has "say so clearly" which is correct but could be stronger
- Capability limitations section is good; it explicitly lists what the bot cannot do

**Improvements needed (to be drafted by planner):**
- Add answer-first rule: "Lead with the answer. Do not open with acknowledgment phrases."
- Reinforce fail fast: "If no tool results match, say so in one sentence. Do not apologize."
- Scope the welcome capability description to specific high-value tasks (trend analysis, finding articles on a specific topic, comparing coverage across sources)

### Anti-Patterns to Avoid

- **Do not add a `@default` to `contentType` or `thinContent`** -- nullable is correct; existing articles have no classification yet
- **Do not reset `enrichedAt` to null for backfill** -- the re-enrichment script should explicitly include all articles, not rely on the `enrichedAt: null` filter in `fetchUnenrichedArticles`
- **Do not use `fetchUnenrichedArticles` in the re-enrichment script** -- it only fetches unenriched articles and has a BATCH_LIMIT. The script needs its own pagination over all articles.
- **Do not delete Dev.to articles separately** -- the `onDelete: Cascade` in `schema.prisma` handles article deletion when the source row is deleted
- **Do not evaluate models before improving the prompt** -- model evaluation must use the improved prompts (D-13 explicitly states this order)

## Common Pitfalls

### Pitfall 1: Re-enrichment using fetchUnenrichedArticles
**What goes wrong:** The developer uses `fetchUnenrichedArticles` from `enrich.ts` for the backfill script. This function filters for `enrichedAt: null` -- so any already-enriched article is skipped. The backfill never runs on existing articles.
**Why it happens:** `fetchUnenrichedArticles` is the obvious import for batch enrichment.
**How to avoid:** The re-enrichment script needs its own `findMany` without the `enrichedAt: null` filter. It should paginate over ALL articles (excluding Dev.to which is already deleted by this point).
**Warning signs:** Script reports "0 articles to process" on first run.

### Pitfall 2: Missing thinContent in saveEnrichmentResults
**What goes wrong:** Schema has `thinContent Boolean?`, Zod schema has `thinContent`, but `saveEnrichmentResults` doesn't include it in the `data` object. thinContent stays NULL after re-enrichment.
**Why it happens:** `thinContent` was already in the Zod schema before the Prisma model column existed. The save function was never updated to match.
**How to avoid:** Add both `contentType` and `thinContent` to the `data` object in `saveEnrichmentResults` simultaneously. Update the `enrich.test.ts` mock assertion to include them.
**Warning signs:** DB shows NULL for `thinContent` after re-enrichment; test passes but DB check fails.

### Pitfall 3: Model string mismatch
**What goes wrong:** Model string in `ai.ts` doesn't exactly match the AI Gateway model ID.
**Why it happens:** Model IDs have exact capitalization and version number requirements.
**How to avoid:** Always copy model IDs verbatim from `curl https://ai-gateway.vercel.sh/v1/models`. The models confirmed in this research: `google/gemini-2.5-flash-lite`, `deepseek/deepseek-v3.2`, `google/gemini-3.1-flash-lite-preview`, `openai/gpt-4.1-mini`.
**Warning signs:** AI SDK throws provider error or 404 at enrichment time.

### Pitfall 4: Seed.ts deleteMany runs on every seed invocation
**What goes wrong:** The Dev.to `deleteMany` preamble in `seed.ts` is a no-op after first run (source already deleted), but if another Dev.to-like source is added later, the pattern might be misread.
**Why it happens:** The preamble pattern from Phase 03.1 used a guard condition (`if (staleHnTldr?.name === "TLDR Tech")`). The Dev.to deletion does not need a guard (deleteMany on a non-existent row is a no-op), but this is worth documenting.
**How to avoid:** `prisma.source.deleteMany({ where: { url: '...' } })` is safe to run multiple times -- it deletes 0 rows if source is gone. No guard needed.

### Pitfall 5: contentType prompt instruction not tight enough
**What goes wrong:** Model assigns `news` to everything because it's the safest default. contentType field becomes useless.
**Why it happens:** Vague taxonomy description in the prompt.
**How to avoid:** Provide concrete distinguishing examples in the enrichment SYSTEM_PROMPT for each contentType. "Announcement" vs "news" is the most common confusion: announcements are from the company itself (press releases, product pages), news is third-party reporting on an event.
**Warning signs:** In evaluation batch, >80% of articles get `news` classification.
