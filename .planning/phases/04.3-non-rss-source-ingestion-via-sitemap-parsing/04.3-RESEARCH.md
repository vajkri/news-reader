## Summary

Both Anthropic News (`anthropic.com/news`) and Claude Blog (`claude.com/blog`) expose `/sitemap.xml` with article URLs. Anthropic's sitemap includes `<lastmod>` dates enabling 7-day cutoff filtering. Claude Blog's sitemap omits `<lastmod>` entirely -- dates must be extracted from the article page itself via `"datePublished"` JSON-LD (confirmed present as `"Sep 10, 2024"` format). Both sites return `og:image`, `og:title`, and `og:description` in page `<head>`, making `ParsedArticle` field extraction straightforward.

Anthropic News dates are embedded in the page as Sanity CMS `_createdAt` fields (ISO format, e.g., `2025-02-18T17:37:28Z`) inside a Next.js RSC payload. Claude Blog uses a Webflow CMS with `"datePublished"` as a human-readable string in a `<script>` block. Both require HTML parsing -- `cheerio` is the right tool for a targeted, low-overhead approach without spinning up a full DOM environment.

Because both target sites have sitemaps with article URLs, both should use the `sitemap` strategy. The `scrape` strategy (listing-page crawl) is scaffolded for future sources without sitemaps. This makes `sitemapPathPattern` the primary config field needed; the scrape strategy additionally needs a `scrapeUrl` (listing page URL) and `scrapeLinkSelector` (CSS selector for article links).

**Primary recommendation:** Use `fast-xml-parser` for sitemap XML, `cheerio` for HTML metadata extraction. Both sources use the `sitemap` strategy. Scaffold `scrape` strategy shell now, implement fully for a future source.

## Source Strategy Decision (CRITICAL)

| Source | Sitemap URL | Has `lastmod`? | Date fallback | Strategy |
|--------|------------|----------------|---------------|----------|
| Anthropic News | `https://www.anthropic.com/sitemap.xml` | YES (ISO 8601) | Sanity `_createdAt` in RSC JSON | `sitemap` |
| Claude Blog | `https://claude.com/sitemap.xml` | NO | `"datePublished"` in Webflow JSON block | `sitemap` |

**Anthropic News** sitemap: 195 `/news/*` entries, with `<lastmod>2025-02-18T17:37:28Z</lastmod>` format. 7-day cutoff applies directly to sitemap `lastmod`.

**Claude Blog** sitemap: 1010+ URLs, `/blog/*` paths, no `<lastmod>`. Date extracted from page body: `"datePublished": "Sep 10, 2024"`. Parse with `Date.parse()` -- reliable for this format. The strategy must fetch each article page to get the date before applying the cutoff.

**Implication for cutoff logic:**
- Sitemap with `lastmod`: filter URLs in the sitemap parse step (cheap, avoids per-page fetches for old articles)
- Sitemap without `lastmod` (claude.com): must fetch each article page to get date, then filter. Apply a secondary sanity check on the page-extracted date. This is expensive -- mitigate by batching fetches with `Promise.allSettled` and a concurrency limit.

### Pattern 1: Strategy Contract (ParsedArticle)
**What:** All three strategies return `ParsedArticle[]`, the interface already defined in `rss.ts`. No changes to interface needed.
**When to use:** Always -- this is the locked contract (D-10).
**Example:**
```typescript
// src/lib/fetch-strategies.ts
import { fetchFeed } from "./rss";
import { fetchSitemap } from "./sitemap";
import { fetchScrape } from "./scrape";
import type { ParsedArticle } from "./rss";

type Source = {
  url: string;
  sourceType: string;
  sitemapPathPattern?: string | null;
  scrapeUrl?: string | null;
  scrapeLinkSelector?: string | null;
};

const strategies: Record<string, (source: Source) => Promise<ParsedArticle[]>> = {
  rss: (source) => fetchFeed(source.url),
  sitemap: (source) => fetchSitemap(source.url, source.sitemapPathPattern ?? ""),
  scrape: (source) => fetchScrape(source.scrapeUrl ?? source.url, source.scrapeLinkSelector ?? "a"),
};

export async function fetchArticles(source: Source): Promise<ParsedArticle[]> {
  const strategy = strategies[source.sourceType] ?? strategies.rss;
  return strategy(source);
}
```

### Pattern 2: Sitemap Strategy
**What:** Parse sitemap XML, filter by path pattern + lastmod cutoff, fetch each article page for metadata.
**When to use:** Sites with `/sitemap.xml` listing article URLs (Anthropic News, Claude Blog).
**Example:**
```typescript
// src/lib/sitemap.ts
import "server-only";
import { XMLParser } from "fast-xml-parser";
import { extractArticleMeta } from "./fetch-article-meta";
import type { ParsedArticle } from "./rss";

const MAX_AGE_MS = 7 * 24 * 60 * 60 * 1000;

export async function fetchSitemap(
  sitemapUrl: string,
  pathPattern: string
): Promise<ParsedArticle[]> {
  const cutoff = Date.now() - MAX_AGE_MS;
  const res = await fetch(sitemapUrl, {
    headers: { "User-Agent": "NewsReader/1.0 (news aggregator)" },
    next: { revalidate: 0 },
  });
  const xml = await res.text();

  const parser = new XMLParser({ ignoreAttributes: false });
  const parsed = parser.parse(xml);
  const urls: Array<{ loc: string; lastmod?: string }> =
    parsed?.urlset?.url ?? [];

  // Filter by path pattern first (cheap)
  const pattern = pathPattern.replace(/\*/g, "");
  const candidates = urls.filter((u) => u.loc?.includes(pattern));

  // Filter by lastmod if present; otherwise must fetch page
  const withLastmod = candidates.filter((u) => u.lastmod);
  const withoutLastmod = candidates.filter((u) => !u.lastmod);

  const recentWithLastmod = withLastmod.filter(
    (u) => new Date(u.lastmod!).getTime() >= cutoff
  );

  // For entries without lastmod: fetch page to get date (claude.com case)
  // Apply concurrency limit to avoid hammering server
  const articles: ParsedArticle[] = [];

  // Process lastmod-filtered entries
  const results = await Promise.allSettled(
    recentWithLastmod.map((u) => extractArticleMeta(u.loc, new Date(u.lastmod!)))
  );
  for (const r of results) {
    if (r.status === "fulfilled" && r.value) articles.push(r.value);
  }

  // Process no-lastmod entries: fetch + check date from page
  if (withoutLastmod.length > 0) {
    const noLastmodResults = await Promise.allSettled(
      withoutLastmod.map((u) => extractArticleMeta(u.loc, null))
    );
    for (const r of noLastmodResults) {
      if (r.status === "fulfilled" && r.value) {
        if (!r.value.publishedAt || r.value.publishedAt.getTime() >= cutoff) {
          articles.push(r.value);
        }
      }
    }
  }

  return articles;
}
```

### Pattern 3: Article Meta Extraction (shared helper)
**What:** Fetches one article page, extracts `og:title`, `og:description`, `og:image`, and publish date using cheerio.
**When to use:** Called by both sitemap and scrape strategies for each discovered URL.
**Example:**
```typescript
// src/lib/fetch-article-meta.ts
import "server-only";
import * as cheerio from "cheerio";
import { estimateReadTime } from "./readtime";
import type { ParsedArticle } from "./rss";

export async function extractArticleMeta(
  url: string,
  lastmod: Date | null
): Promise<ParsedArticle | null> {
  try {
    const res = await fetch(url, {
      headers: { "User-Agent": "NewsReader/1.0 (news aggregator)" },
      signal: AbortSignal.timeout(15000),
    });
    if (!res.ok) return null;
    const html = await res.text();
    const $ = cheerio.load(html);

    const title =
      $('meta[property="og:title"]').attr("content") ??
      $("title").text().trim() ??
      "Untitled";
    const description =
      $('meta[property="og:description"]').attr("content") ??
      $('meta[name="description"]').attr("content") ??
      null;
    const thumbnail =
      $('meta[property="og:image"]').attr("content") ??
      $('meta[name="twitter:image"]').attr("content") ??
      null;

    // Extract date: JSON-LD datePublished, or article:published_time, or fall back to lastmod
    let publishedAt: Date | null = lastmod;
    const scriptContent = $('script[type="application/ld+json"]').text();
    const dateMatch = scriptContent.match(/"datePublished"\s*:\s*"([^"]+)"/);
    if (dateMatch) {
      const parsed = new Date(dateMatch[1]);
      if (!isNaN(parsed.getTime())) publishedAt = parsed;
    }
    if (!publishedAt) {
      const metaDate = $('meta[property="article:published_time"]').attr("content");
      if (metaDate) {
        const parsed = new Date(metaDate);
        if (!isNaN(parsed.getTime())) publishedAt = parsed;
      }
    }

    return {
      guid: url, // URL as guid (stable, deduplicates re-ingestion)
      title,
      link: url,
      description,
      thumbnail,
      publishedAt,
      readTimeMin: estimateReadTime(description),
    };
  } catch {
    return null;
  }
}
```

### Anti-Patterns to Avoid
- **Using `rss-parser` for sitemap XML:** `rss-parser` is for Atom/RSS document schemas; sitemaps use `<urlset>/<url>/<loc>/<lastmod>` which is a different schema. Use `fast-xml-parser`.
- **Fetching all sitemap URLs for date-only-known sources:** Claude Blog has 1010+ sitemap entries. Fetching all pages to check dates is expensive. Apply a URL-based heuristic if available (e.g., if slug contains a year), or cap batch size.
- **`new Date("Sep 10, 2024")`:** Works in Node, but mark it explicitly -- V8 parses this correctly, but it's not ISO 8601. Log a warning if parse returns NaN, fall back to null.
- **Using `source.url` as sitemap URL for the display link:** The `source.url` is now the sitemap endpoint. The article link must come from `<loc>` in the sitemap, not `source.url`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XML parsing | Custom string regex for `<loc>` and `<lastmod>` | `fast-xml-parser` | Regex breaks on namespaces, whitespace variations, CDATA; `fast-xml-parser` handles all edge cases |
| HTML meta extraction | `string.match(/<meta property="og:title"/)` | `cheerio` | Attribute order, encoding, self-closing tags are all traps; cheerio handles them |
| Concurrency limiting | Custom Promise queue | `Promise.allSettled` with batching | Sufficient for 7-day cadence; article count per run is low (cron is daily) |

**Key insight:** The 7-day cutoff dramatically limits article volume per run. Anthropic News has ~195 total entries across all time; recent week is typically 5-15. Claude Blog is larger but same principle. There is no need for sophisticated concurrency management -- `Promise.allSettled` is fine.

## Common Pitfalls

### Pitfall 1: Claude Blog Sitemap Has No `lastmod`
**What goes wrong:** The 7-day cutoff logic skips entries with no `lastmod`, resulting in zero Claude Blog articles ingested.
**Why it happens:** Claude Blog (Webflow-based) does not emit `<lastmod>` in sitemap.
**How to avoid:** When `lastmod` is absent, fetch the article page and parse `"datePublished"` from the Webflow JSON block (`<script>` containing `"datePublished": "Sep 10, 2024"`).
**Warning signs:** No Claude Blog articles appearing after first run despite populated sitemap.

### Pitfall 2: Claude Blog Page 404s for Old URL Patterns
**What goes wrong:** Some claude.com/blog URLs in the sitemap return 404 (confirmed during research: `claude.com/blog/claude-code` returns 404, only certain slugs are live).
**Why it happens:** Webflow CMS may have published/unpublished pages or redirected URLs.
**How to avoid:** Handle non-200 responses gracefully in `extractArticleMeta` -- return null, skip silently. Do not error the whole source on individual article 404s.
**Warning signs:** High error rate on Claude Blog source in cron response.

### Pitfall 3: Anthropic News og:image Uses an API Endpoint
**What goes wrong:** `og:image` on Anthropic News is `https://www.anthropic.com/api/opengraph-illustration?name=...&backgroundColor=clay` -- a dynamic URL, not a static image CDN path.
**Why it happens:** Anthropic generates OG images server-side.
**How to avoid:** This is actually fine -- the URL is valid and will render in the UI. No special handling needed. Just store the URL as-is.
**Warning signs:** Broken images in UI if Anthropic changes the API endpoint.

### Pitfall 4: Sitemap Source `url` Field Semantics Change
**What goes wrong:** Existing sources store the RSS feed URL in `Source.url`. New sitemap sources store the sitemap URL. The fetch cron previously called `fetchFeed(source.url)` -- after refactor, `source.url` is the sitemap URL for sitemap sources.
**Why it happens:** Overloading the `url` field to mean different things per `sourceType`.
**How to avoid:** The strategy dispatcher handles this -- `fetchFeed` receives `source.url` (RSS feed), `fetchSitemap` receives `source.url` (sitemap URL). Document this convention in source code comments.
**Warning signs:** RSS sources breaking if the dispatch logic passes the wrong URL to the wrong strategy.

### Pitfall 5: Prisma Type Mismatch After Schema Change
**What goes wrong:** TypeScript errors in `fetch/route.ts` because the Prisma `Source` type doesn't include `sourceType` etc. until `prisma generate` runs.
**Why it happens:** Prisma generates types from schema; must regenerate after migration.
**How to avoid:** Run `pnpm prisma migrate dev --name add-source-type-fields` then `pnpm build` to verify types.
**Warning signs:** `Property 'sourceType' does not exist on type 'Source'` TypeScript errors.

### Pitfall 6: `guid` Collision on Re-ingestion
**What goes wrong:** Using URL as `guid` is safe for deduplication, but if a site changes URL (e.g., redirects) the same article gets re-ingested with a new guid.
**Why it happens:** URL-based guid depends on URL stability.
**How to avoid:** Use the article URL from `<loc>` (canonical, not redirect destination). For Phase 04.3, URL-based guids are sufficient -- the lock decision is made (D-12).
**Warning signs:** Duplicate articles from the same source in the feed.

## Scrape Strategy Config Fields (Claude's Discretion - D-12)

The scrape strategy (for future sources without RSS or sitemaps) fetches a listing page and extracts article links. Two fields are needed on the Source model:

| Field | Type | Purpose | Example |
|-------|------|---------|---------|
| `scrapeUrl` | `String?` | The listing page URL to fetch | `https://some.blog/articles` |
| `scrapeLinkSelector` | `String?` | CSS selector targeting article link elements | `"article h2 a"` or `".post-list a"` |

Rationale: a listing page (distinct from `source.url` which is the sitemap URL for `sitemap` sources) plus a CSS selector is the minimal viable config for scraping. Adding `scrapePathPattern` (to filter extracted links) is optional enhancement -- skip for Phase 04.3, add when a real scrape source is configured.

The scrape strategy should be scaffolded (function signature, `ParsedArticle[]` return, `not implemented` error or empty return) but not fully implemented unless time permits. First real source needing it will drive the implementation.

## `guid` Generation Decision

**Recommendation: URL as guid** (not content hash).

- URL is stable for Anthropic News and Claude Blog (both use canonical slugs)
- Matches how RSS `guid` often equals the article link
- Simple, deterministic, debuggable
- Risk: URL changes = duplicate. Acceptable for this corpus.

Implementation: `guid = article URL from <loc>` (already the `link` field).

## Open Questions

1. **Claude Blog: fetch cost for date extraction**
   - What we know: ~1010 sitemap entries total; new entries per week is low (estimated 3-10)
   - What's unclear: No way to pre-filter without `lastmod`; all new-to-db URLs require a page fetch
   - Recommendation: On first run, cap at 50 most recently added sitemap URLs (tail of list). Use URL-based heuristic -- the sitemap appears ordered; new posts likely near the end. On subsequent runs, guid deduplication prevents re-processing.

2. **Anthropic article body content**
   - What we know: Anthropic News pages are Next.js with Sanity RSC payloads. The article body is in the RSC JSON (`_type: "post"`, `body` array of portable text blocks), not in a standard `<article>` tag visible to cheerio without JS rendering.
   - What's unclear: How rich is `og:description`? Is it sufficient for enrichment?
   - Recommendation: Use `og:description` for Phase 04.3. It maps to the same signal RSS `description` provides. If enrichment quality is poor, add Readability-based extraction in a follow-up phase.

3. **Rate limiting on target sites**
   - What we know: Both sites serve public content. No robots.txt block found for aggregators.
   - What's unclear: Will rapid parallel fetches trigger rate limiting or temporary IP blocks?
   - Recommendation: Add 100ms delay between article page fetches per source (not across sources). `Promise.allSettled` with sequential processing of each source's articles is safe; article count per 7-day window is small.
