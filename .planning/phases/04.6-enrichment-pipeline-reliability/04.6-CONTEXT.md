# Phase 04.6: Enrichment Pipeline Reliability - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the enrichment pipeline handle daily article volume reliably and redesign the briefing model so users can trust they are not missing important news. This phase delivers: a new "catch me up" briefing model with watermark tracking, GitHub Actions-based scheduling for fetch and enrich, on-demand manual enrichment from the briefing page, pending article visibility, increased batch size, newest-first ordering, immutable archive briefings, and user feedback controls for importance scoring calibration.

</domain>

<decisions>
## Implementation Decisions

### Briefing Model
- **D-01:** Default briefing mode is "catch me up": shows all enriched articles since the user's last-read watermark, not a fixed daily window
- **D-02:** Watermark model for new/reviewed state: a single timestamp per user tracks last visit; articles enriched before watermark = reviewed (dimmed), after = new (with "New" badge)
- **D-03:** Watermark updates on briefing page visit (not on explicit "mark as read" button)
- **D-04:** "All caught up" empty state when no new articles exist since last watermark, with today's full briefing collapsed below for reference
- **D-05:** Archive mode via DateStepper shows frozen top 15 articles by importance for that date, regardless of how many enrichment runs happened. Archive has no feedback buttons or "Enrich now" button. Clear "Archive" banner distinguishes it from live view.

### Enrichment Triggers
- **D-06:** GitHub Actions workflow replaces Vercel cron jobs. Single workflow file runs every 4 hours, calling /api/fetch then /api/enrich sequentially. CRON_SECRET stored as GitHub repo secret.
- **D-07:** Remove both cron entries from vercel.json entirely
- **D-08:** On-demand enrichment on briefing page: when unenriched articles exist, show them in an amber "pending enrichment" section with an "Enrich now (N)" button. User clicks to trigger, page reloads with results. No auto-enrichment on page load.

### Batch Sizing and Throughput
- **D-09:** BATCH_LIMIT increases from 15 to 25. Tested: DeepSeek v3.2 handles 50 articles in 113s; 25 keeps each call under 60s
- **D-10:** Keep maxDuration = 60 on the enrich route. The existing time-budget loop handles multiple iterations within the budget.
- **D-11:** Enrichment scope: only articles fetched recently (not ancient backlog). If an article wasn't enriched within 48h of being fetched, it's abandoned as stale.

### Ordering and Prioritization
- **D-12:** fetchUnenrichedArticles changes from publishedAt ASC (oldest first) to publishedAt DESC (newest first), so the most recent articles are always enriched first

### Immutability
- **D-13:** Re-enrichment must not change past briefings. Archive view queries by publishedAt date, not enrichedAt, so re-enriching an article doesn't move it between days.

### User Feedback (Learning Period)
- **D-14:** Each article in the "catch me up" view has up/down arrow buttons for importance feedback. This feedback is stored for future enrichment calibration. Feedback controls only appear on current articles, not archive.

### Claude's Discretion
- Schema design for watermark storage (cookie, localStorage, or DB field)
- Schema design for user feedback storage
- Exact 48h staleness cutoff implementation
- Whether "Enrich now" calls the existing /api/enrich endpoint or a separate on-demand endpoint
- Status bar design details (last enriched timestamp, new count format)

### Folded Todos
- **Restore briefing page caching**: STATE.md notes "consider at end of 04.6." With the new "catch me up" model using force-dynamic, caching strategy needs to be reconsidered as part of this phase.

</decisions>

<specifics>
## Specific Ideas

- User explicitly liked the "pending enrichment" amber section in the on-demand mockup: articles shown with title and "Awaiting enrichment" label, dimmed, with amber left border
- Morning/evening visit split: "New since [time]" section on top with full cards, "Already reviewed" section below with dimmed collapsed cards
- "All caught up" state should feel reassuring: checkmark icon, clear messaging, collapsed reference to today's articles below
- Archive banner should clearly say "Archive: [date]" to distinguish from live view
- User checks news morning and evening; the "catch me up" model directly serves this workflow

</specifics>

<deferred>
## Deferred Ideas

- **Per-article read tracking on briefing**: Watermark model is sufficient for now. Per-article "I read this" tracking could be added later if the watermark proves too coarse.
