| Role | CSS Token | Light value | Dark value | Usage |
|------|-----------|-------------|------------|-------|
| Dominant (60%) | `--background` | #ffffff | #09090b | Page background; no change |
| Secondary (30%) | `--muted` / `--secondary` | #f4f4f5 | #27272a | Debug annotation block backgrounds; DebugToggle outline hover |

**Accent reserved for (this phase):** DebugToggle button when debug mode is ON (uses `variant="outline"` at 60% opacity normally; full opacity on hover — `--primary` border/ring confirms active state visually).

**Debug annotation color contract:**
- Background: `bg-(--muted)` — secondary surface, not dominant
- Text (metadata values): `text-(--muted-foreground)` — zinc-500/zinc-400
- Text (field labels): `text-(--foreground-secondary)` — zinc-700 light / zinc-300 dark
- Border (JSON pre block separator): `border-(--border)`
- JSON pre block: `bg-(--muted) text-(--muted-foreground)` — same muted surface, subdued palette confirms dev-only status

**DebugToggle visual states:**

| State | Visual |
|-------|--------|
| Debug OFF | `variant="outline"` `opacity-60` — unobtrusive, recedes |
| Debug ON | `variant="outline"` `opacity-100` — full presence, visually distinct |
| Hover (either state) | `opacity-100` — hover always reveals |

## Interaction Contract

### DebugToggle button

- **Position:** `fixed bottom-4 right-4 z-50`
- **Default state:** opacity 60%, `Bug` icon (debug OFF)
- **Active state:** opacity 100%, `BugOff` icon (debug ON)
- **On click:** toggles `debugMode` boolean in `DebugContext`
- **Keyboard:** inherits `focus-visible:` outline from `Button` component — no additional focus styling needed
- **aria-label:** `"Enable debug mode"` when OFF; `"Disable debug mode"` when ON
- **aria-pressed:** `true` when debug ON, `false` when OFF
- **Production guard:** Returns `null` if `process.env.NODE_ENV !== 'development'`

### BriefingDebugBox

- **Renders:** only when `debugMode === true` (reads from `useDebug()`)
- **Position:** between the briefing page heading and the first topic group, in normal document flow
- **Layout:** single block, no collapse/expand
- **Content (from CONTEXT.md D-07):**
  - Timeframe: `windowStart` to `windowEnd` formatted as human-readable date range
  - Article count in the current briefing window
  - Latest `enrichedAt` timestamp (proxy for enrich cron run time)
  - No separate EnrichRun table required — article-derived metadata only

### Per-card debug JSON (BriefingCard)

- **Renders:** only when `debugMode === true`
- **Position:** below the card content, attached to the card element as a `<pre>` block
- **Collapse/expand:** none — always fully visible when debug mode ON (CONTEXT.md D-09)
- **Content:** `JSON.stringify(article, null, 2)` — full article record including all fields fetched by the briefing query
- **Note on `contentType` / `thinContent`:** Include these fields. Add to `ArticleRow` type and briefing Prisma `include`. Dev-only query; cost is irrelevant. Makes JSON complete.

### State management (D-04 resolution)

React Context (`DebugProvider` + `useDebug` hook) over HTML `data-*` attribute pattern.

**Rationale (from RESEARCH.md):**
- `BriefingCard` is `'use client'` — can consume Context directly
- `BriefingPage` is a Server Component — cannot read `document` attributes at all
- `data-chat-open` was chosen for CSS `:has()` grid reflows; debug mode has no CSS-only consumers
- Context is tree-scoped and React-native; no `document.documentElement` coupling

## Copywriting Contract

This phase is dev-only infrastructure. No user-facing copy is introduced. All strings below are developer-facing only and appear exclusively in `NODE_ENV === 'development'`.

| Element | Copy | Notes |
|---------|------|-------|

| DebugToggle aria-label (ON) | `Disable debug mode` | Accessible label; dev-only |

| BriefingDebugBox article count label | `Articles in window` | Field label prefix |
| BriefingDebugBox enrich timestamp label | `Latest enrichedAt` | Field label prefix |
| Empty state | N/A | No empty state — debug box always has data when briefing has articles |
| Error state | N/A | No error state — debug is passive read-only display |
| Destructive confirmation | N/A | No destructive actions in this phase |

## Production Footprint

Zero. All debug UI is gated behind `process.env.NODE_ENV !== 'development'` checks at two levels:
1. `DebugProvider` — returns `<>{children}</>` unchanged in production (no state, no context value)
2. `DebugToggle` — returns `null` in production
3. Next.js statically analyzes `process.env.NODE_ENV` and tree-shakes dead branches in production bundles

The production bundle includes no debug component code, no debug state, no debug DOM nodes.

## Accessibility

- `Button` component handles `focus-visible:` outline — no additional work needed
- `aria-pressed` on DebugToggle communicates toggle state to screen readers
- `aria-label` on DebugToggle (icon-only button) provides accessible name
- Debug `<pre>` blocks: monospace, `overflow-x-auto` — no horizontal scroll trap
- Contrast: `text-(--muted-foreground)` on `bg-(--muted)` — verified in existing design system; debug-only surface uses the same tokens as existing muted surfaces throughout the app
- Debug components are dev-only and never reach production; WCAG AA applies to all production-visible states only
