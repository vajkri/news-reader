# Phase 04.5: Dev Debug Mode Infrastructure - Research

**Researched:** 2026-03-28
**Domain:** React dev tooling, Next.js conditional rendering, client-side state management
**Confidence:** HIGH

## Summary

This phase adds a dev-only debug toggle and inline annotations to the briefing page. The entire feature must disappear at zero cost in production builds (`NODE_ENV === 'production'`). The main design decision (D-04) is whether to use React Context or an HTML data attribute for the global boolean toggle state.

The codebase already has two patterns to draw from: `NODE_ENV` checks in `scrape.ts` and `prisma.ts` for dev-only behavior, and the `data-chat-open` HTML attribute on `<html>` for decoupled cross-component coordination in `ChatWrapper`. The briefing page is a Server Component, which constrains how debug state can flow down to `BriefingCard` (a `'use client'` component).

**Primary recommendation:** Use React Context (`DebugProvider` + `useDebug` hook) over the HTML data attribute pattern. The HTML attribute pattern suits CSS-driven layout shifts (like the chat grid); the debug mode needs boolean state accessible in React component trees without `document` coupling. React Context is the idiomatic choice for a dev-only app-wide boolean that multiple components need to read.

### Pattern 1: `DebugProvider` + `useDebug` hook

**What:** A React Context that holds a `debugMode: boolean` and a `toggleDebug` function. All debug UI reads from `useDebug()` and conditionally renders.

**When to use:** When multiple components at different tree depths need the same boolean flag without prop drilling.

**Why React Context over data attribute (D-04 recommendation):**
- `BriefingCard` is a `'use client'` component — it can consume Context directly
- `BriefingPage` is a Server Component — it cannot read `document` attributes at all
- Context is tree-scoped and React-native; no `document.documentElement` coupling
- `data-chat-open` was chosen specifically because CSS grid reacts to it via `:has()` — debug mode has no CSS-only consumers

**Example:**
```typescript
// src/contexts/debug.tsx
'use client';

import { createContext, useContext, useState } from 'react';

interface DebugContextValue {
  debugMode: boolean;
  toggleDebug: () => void;
}

const DebugContext = createContext<DebugContextValue>({
  debugMode: false,
  toggleDebug: () => {},
});

export function DebugProvider({ children }: { children: React.ReactNode }) {
  // Only render in development — zero footprint in production
  if (process.env.NODE_ENV !== 'development') {
    return <>{children}</>;
  }

  return <DebugProviderInner>{children}</DebugProviderInner>;
}

function DebugProviderInner({ children }: { children: React.ReactNode }) {
  const [debugMode, setDebugMode] = useState(false);
  return (
    <DebugContext.Provider value={{ debugMode, toggleDebug: () => setDebugMode((v) => !v) }}>
      {children}
    </DebugContext.Provider>
  );
}

export function useDebug(): DebugContextValue {
  return useContext(DebugContext);
}
```

```typescript
// src/components/features/debug/DebugToggle.tsx
'use client';

import { useDebug } from '@/contexts/debug';
import { Button } from '@/components/ui/button';
import { Bug, BugOff } from 'lucide-react';

export function DebugToggle() {
  if (process.env.NODE_ENV !== 'development') return null;

  const { debugMode, toggleDebug } = useDebug();

  return (
    <Button
      variant="outline"
      size="icon"
      onClick={toggleDebug}
      className="fixed bottom-4 right-4 z-50 opacity-60 hover:opacity-100"
      aria-label={debugMode ? 'Disable debug mode' : 'Enable debug mode'}
      aria-pressed={debugMode}
    >
      {debugMode ? <BugOff size={16} /> : <Bug size={16} />}
    </Button>
  );
}
```

### Pattern 2: Server Component data flow for `BriefingDebugBox`

**What:** The briefing page already computes `windowStart`/`windowEnd` for the date filter. Pass these directly as props to a client component that only renders when `useDebug().debugMode` is true.

**Key insight:** The enrich cron (`/api/enrich`) does not currently persist per-run metadata (model used, batch size, timestamp) to the database. The debug box can show `enrichedAt` timestamps from the articles themselves as a proxy — e.g. the most recent `enrichedAt` in the current briefing window — rather than requiring a new DB table. This keeps scope minimal.

**Example:**
```typescript
// In briefing/page.tsx (Server Component) — pass debug data via props
<BriefingDebugBox
  windowStart={windowStart.toISOString()}
  windowEnd={windowEnd.toISOString()}
  articleCount={articles.length}
  latestEnrichedAt={articles.reduce((latest, a) =>
    a.enrichedAt && (!latest || a.enrichedAt > latest) ? a.enrichedAt : latest
  , null as string | null)}
/>
```

```typescript
// src/components/features/briefing/BriefingDebugBox.tsx
'use client';
import { useDebug } from '@/contexts/debug';

interface Props {
  windowStart: string;
  windowEnd: string;
  articleCount: number;
  latestEnrichedAt: string | null;
}

export function BriefingDebugBox(props: Props) {
  const { debugMode } = useDebug();
  if (!debugMode) return null;
  // render timeframe + enrichment metadata
}
```

### Pattern 3: Per-card debug JSON

**What:** `BriefingCard` receives its `article` prop. When debug mode is ON, render a `<pre>` block below the card with the full article JSON.

**Key insight:** `BriefingCard` already has `'use client'` — it can call `useDebug()` directly. The `article` prop (`BriefingArticle`) already contains all available DB fields (from `ArticleRow`): id, guid, title, link, description, thumbnail, publishedAt, readTimeMin, isRead, summary, topics, importanceScore, enrichedAt, contentType, thinContent, sourceId, source. Note `contentType` and `thinContent` exist in the Prisma schema but are NOT currently in `ArticleRow` or included in the briefing page query — the plan must add them to the Prisma `include` and `ArticleRow` type if they should appear in debug output.

**Example:**
```typescript
// Inside BriefingCard, after the existing JSX
{debugMode && (
  <pre
    className="text-[0.75rem] font-[family-name:var(--font-geist-mono)] bg-(--muted) text-(--muted-foreground) p-3 overflow-x-auto rounded-b-[0.625rem] border-t border-(--border)"
  >
    {JSON.stringify(article, null, 2)}
  </pre>
)}
```

### Anti-Patterns to Avoid
- **Calling `useDebug()` in Server Components:** `DebugProvider` is `'use client'`; Server Components cannot access React Context. All debug state reads must be in client components.
- **Using `process.env.NODE_ENV` checks inside React component bodies expecting tree-shaking:** Next.js does tree-shake `NODE_ENV !== 'development'` branches in production bundles, but the `DebugProvider` wrapper approach is safer — the inner state component only mounts in dev.
- **Adding `contentType`/`thinContent` to debug output without updating the Prisma query:** The briefing page `include` currently only selects `source.name` and `source.category`. If the plan wants those fields in the JSON, they need to be fetched.
- **`data-debug-mode` on `<html>` pattern:** Would require `document.documentElement` reads inside useEffect, making SSR/hydration synchronization error-prone.

## Common Pitfalls

### Pitfall 1: DebugProvider placement breaks SSR hydration
**What goes wrong:** Placing `DebugProvider` inside a Server Component wrapper that conditionally renders based on `NODE_ENV` can cause React hydration mismatches if the client-side value differs from server-rendered HTML.
**Why it happens:** Server renders with one `NODE_ENV` value; if the check is evaluated at different times, or if the component tree changes shape between server and client, React throws hydration errors.
**How to avoid:** Keep `DebugProvider` in `layout.tsx` unconditionally; do the `NODE_ENV` check inside the Provider to short-circuit rendering the inner state component, not to conditionally mount the Provider itself.
**Warning signs:** "Hydration error: server rendered X, client rendered Y" in dev console.

### Pitfall 2: `useDebug()` called in a Server Component
**What goes wrong:** Calling `useDebug()` in `briefing/page.tsx` (a Server Component) throws a build error.
**Why it happens:** `page.tsx` has no `'use client'` directive; React Context is client-only.
**How to avoid:** All debug state reads go in `'use client'` components (`BriefingCard`, `BriefingDebugBox`, `DebugToggle`). The Server Component only passes data props.
**Warning signs:** Build error: "createContext is not defined" or "Cannot use hooks in Server Components".

### Pitfall 3: Missing fields in article debug JSON
**What goes wrong:** The per-card JSON only shows fields in `BriefingArticle` (which extends `ArticleRow`), but `ArticleRow` omits `contentType` and `thinContent` — schema fields added in Phase 04.1 that were never added to the shared type.
**Why it happens:** `ArticleRow` in `src/types/index.ts` was defined before Phase 04.1 added those columns. The briefing page `include` also doesn't select them.
**How to avoid:** Either (a) add `contentType` and `thinContent` to `ArticleRow` and update the briefing query's `include`, or (b) accept that those fields won't show in debug JSON and document the limitation. Decision for the planner.
**Warning signs:** Debug JSON shows `undefined` or fields are absent where Prisma schema has values.

### Pitfall 4: `DebugToggle` visible in production builds
**What goes wrong:** The floating button appears in production if `NODE_ENV` check is forgotten or component is imported unconditionally.
**Why it happens:** Next.js only tree-shakes dead code when the condition is statically analyzable at build time.
**How to avoid:** Both the `DebugProvider` content and `DebugToggle` must check `process.env.NODE_ENV !== 'development'` and return `null`/`children` early. Additionally, avoid dynamic imports without `{ ssr: false }` for debug components — they can still be included in the bundle.
**Warning signs:** Debug button visible on production URL; bundle analyzer shows debug components in production chunks.

## Open Questions

1. **Should `contentType` and `thinContent` be included in debug JSON?**
   - What we know: Fields exist in Prisma schema; not currently in `ArticleRow` or briefing query
   - What's unclear: Whether the extra query cost is worth it for a dev-only feature
   - Recommendation: Add them. It's a dev-only query; cost is irrelevant. Add both fields to `ArticleRow`, update the briefing `include`, update the briefing Prisma query. Makes debug JSON complete.

2. **What enrich cron metadata is available for D-07?**
   - What we know: The enrich route returns `{ enriched: number, errors: string[] }` but this is not persisted. No separate `EnrichRun` table exists.
   - What's unclear: Whether D-07 expects live per-run metadata or article-derived metadata
   - Recommendation: Use article-derived metadata: show `windowStart`/`windowEnd` (already computed), total article count, `importanceScore` distribution, and the range of `enrichedAt` timestamps within the window. This requires no schema change. The plan should document this scope decision explicitly.
